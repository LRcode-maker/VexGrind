01 Sensor Calibration

Before any maze logic could be written, we needed to understand exactly how the bumper sensor was measuring distance. We set up a simple test: used a tape measure and a short script that would back the robot up 600 mm from the moment the bumper triggered.

What we quickly discovered was that the sensor wasn't measuring from the robot's centre — it was measuring until it hit 600 mm from the rear of the bumper itself. A subtle but critical distinction that would have broken all our subsequent code if we'd missed it.

"The bumper measured to 60 cm from the rear — not the centre. One small detail, massive downstream impact."
🔬 Measurement Insight

Always verify sensor reference points before writing motion logic. The bumper triggers from its physical contact surface, not from the robot's geometric centre — offset accordingly in your drive constants.

02 The First Exercise

With the sensor understood, we set ourselves a starter challenge: make the robot loop endlessly inside a box using only the bumper. No complex pathfinding — just drive forward, hit a wall, back up, turn, repeat.

This took minimal code to produce, and it worked — but we ran into an immediate real-world problem. Wheel traction wasn't perfectly consistent, which meant over many cycles the robot would gradually drift out of alignment. The box loop would slowly degrade into a spiral.

It was a great first exercise precisely because it surfaced a hardware constraint early — one that would shape every design decision going forward.

  • Win → Proved bumper-only autonomy works in a closed environment with minimal code.
  • Problem → Traction inconsistency caused cumulative drift over many loops, breaking the pattern.
  • Lesson → Random turning angles actually helped — drift never accumulated in a single direction.

03 The Problem

Next, we escalated the challenge: escape a hexagonal box where the only exits were at the corners, and the robot had to use random turning to find them. No wall-mapping, no memory — pure stochastic navigation.

The robot was clearly capable of escaping — it had the speed and the sensor precision. The question was how long it would take and whether the random turn strategy would be statistically reliable enough across different starting positions.

04 The Robot Code Process

The logic is deceptively simple — three repeating phases that together produce reliable maze-escape behaviour without any map, memory, or complex decision tree.

  • 1

    Drive Forward

    Normal operation. The robot moves forward at constant speed until the bumper sensor is triggered by a wall contact.

  • 2

    Wait Until Bumper Pressed → Back Up 600 mm

    When the bumper triggers, the robot immediately reverses 600 mm (corrected for our calibrated sensor offset) to clear the wall.

  • 3

    Random Turn

    The robot turns left by a randomly-generated angle. This stochastic step prevents it from repeatedly driving into the same wall at the same angle, ensuring eventual escape through any gap.

// Repeat forever — bump, back, turn randomly
while (true) {
  Drivetrain.drive(forward);
  waitUntil(Bumper.pressing());

  Drivetrain.driveFor(reverse, 600, mm);

  int turnDeg = random(30, 150);
  Drivetrain.turnFor(left, turnDeg, degrees);
}
⚡ Key Takeaway

Randomness is a legitimate navigation strategy. By removing determinism from the turn phase, the robot avoids all cyclic trap states — it will always eventually find an exit, no matter the arena shape. The simpler the logic, the more robust the behaviour in unpredictable environments.
