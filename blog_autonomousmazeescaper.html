We Built a VEX V5 Robot That
Escapes Any Maze.

Full breakdown of our autonomous maze-solving robot — the sensor stack, the wall-following algorithm, the dead-end recovery logic, and everything we learned building it from scratch.

At first we needed to test how the sensor was measuring distance. We achieved this small feat by using a tape measure and some simple code that backed to robot to 60cm behind the spot it hit (the bumper). We figured out that the bumper was measuring until it hit 60 cm not the rear.

The Exercise

Using this knowlege we constructed a challenge for us: to make the robot endlessly in a box with the bumper. A fairly simple starter challenge. This took minimal effort to produce however their were slight problems with traction from the wheels eventually misalligning it.

This simple challenge was a great exercise as it helped to push our understanding of the bumper sensor and code.

The Problem

Our next problem was to escape a hexagon box with escape points at each corner but we had to use random turning. The robot was clearly capable of beating it

The Robot Thing process

  • 1

    FOLLOW_LEFT

    Normal operation. Left sensor reads a wall at 80–140mm. Robot drives forward while a P-controller keeps that distance constant. Any deviation triggers a proportional correction to steering.

  • 2

    TURN_LEFT

    Left sensor drops below threshold (open space to the left). Robot completes its forward stride to round the corner fully, then executes a 90° left turn using the IMU for precision heading.

  • 3

    TURN_RIGHT

    Front sensor detects wall within 150mm AND left sensor detects wall. Robot stops, turns right 90°. If front and left and right are all blocked — triggers state 4.

  • 4

    REVERSE

    Dead-end detected. Robot turns 180° using two sequential 90° turns (more reliable than a single 180° on the V5 IMU), re-enters the FOLLOW_LEFT state.

💡 Tuning Tip

The KP value of 0.35 was found after about 40 test runs. Too high and the robot oscillates down corridors like a drunk. Too low and it drifts into walls before correcting. Start at 0.2, increase until you see oscillation, then back off 30%.

What Went Wrong (And How We Fixed It)

Nothing ever works first try. Here's the shortlist of problems we hit and the fixes that actually worked:

  • IMU drift on long runs — After 10+ turns the heading was off by 3–5°. Fixed by re-zeroing the IMU every time we detected a stable straight corridor using the wall sensors as a sanity check.
  • Ghost readings at corners — Distance sensors would misread open diagonal space at corners as a clear path too early. Fixed with a 3-sample moving average and a 30ms debounce before acting on any open-space detection.
  • Wheel slip on turns — Fast in-place turns caused the robot to overshoot its target heading. Slowed turn speed to 40% and the IMU accuracy went from ±5° to ±1°.
  • Dead-end oscillation — Early builds would sometimes detect a dead-end, start reversing, then re-detect the same dead-end mid-turn. Fixed by adding a TURNING state that ignores sensor input until the manoeuvre completes.

Results

After 30 runs across three different maze configurations, the robot achieved a 100% completion rate with an average escape time of 4.2 seconds. Zero wall collisions after the sensor debounce fix went in.

✅ Outcome

The robot handles T-junctions, dead ends, L-turns, and long straight corridors reliably. The only maze type it can't solve is one with loops — the left-hand rule will circle forever. That's a Flood Fill problem for another day.

What's Next

We're planning to extend this into a full maze mapper using encoder odometry — the robot will build a grid representation of the maze as it explores, then use Flood Fill to find the shortest path on a second pass. Faster, smarter, and applicable directly to multi-zone autonomous strategies in competition.
